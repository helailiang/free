# 单线激光雷达 C2/H1 网络通信测试方案（整理版）

## 1. 文档目的

本文基于《单线激光雷达（C2H1）网络通信测试方案.docx》整理，用于指导 C2/H1 单线激光雷达网络通信能力的验证、自动化测试开发和现场测试执行。

整理后的重点是把原方案从“测试想法集合”收敛为“可执行测试方案”，让测试人员能按模块准备环境、执行用例、记录结果，并为后续 Python 自动化测试框架开发提供依据。

## 2. 测试目标

本次测试需要验证雷达在真实工程环境中的网络通信可靠性，主要覆盖以下目标：

1. 验证雷达从物理网口到应用层协议的基础通信能力。
2. 验证网络延迟、丢包、抖动、断线、IP 冲突、背景流量等异常情况下的恢复能力。
3. 验证 C2/H1 在长期运行、多设备组网、动态移动场景下的数据连续性。
4. 验证上位机、ROS 驱动、Python 自动化脚本对雷达数据接收和统计的稳定性。
5. 建立可量化的测试准入标准，为量产、交付和现场排障提供依据。

## 3. 测试范围

本方案建议按 7 个模块执行：

1. 硬件与连接测试：电源适应性、网口插拔、线缆长度、基础 ping。
2. 通信协议与配置测试：IP 配置、设备发现、指令响应、数据格式、多客户端连接。
3. 网络健壮性测试：延迟、抖动、丢包、乱序、断线恢复、IP 冲突、背景流量。
4. 整机集成与动态测试：移动小车、转台、多雷达、多传感器共网。
5. 可靠性与安全测试：长稳、固件升级、端口扫描、协议模糊测试。
6. ROS 驱动测试：点云接收、异常断连、重连、长时间取数。
7. 自动化测试框架：Python + Pytest 自动执行、统计、报告生成。

## 4. 测试通过标准

### 4.1 基础准入标准

1. 局域网基础延迟：C2/H1 平均延迟建议小于 2 ms；若实际网络拓扑包含交换机、损伤仪或 USB 网卡，需要单独记录测试环境。
2. 协议响应正确率：标准指令、标准数据流解析正确率应为 100%。
3. 基础连通性：长时间 ping 成功率不低于 99.9%。
4. 断线恢复：C2 建议不超过 10 秒，H1 建议不超过 5 秒。
5. 动态场景帧率波动：移动速度 1.5-4 m/s 时，帧率波动建议控制在 ±10% 以内。
6. 长期稳定性：72 小时基础长稳不得出现程序崩溃、设备死机、连接泄漏、明显内存持续增长。

### 4.2 压力准入标准

1. 在 1% 丢包 + 100 ms 延迟的网络损伤条件下，C2 应用层数据丢包率建议不高于 1%。
2. 在 1% 丢包 + 100 ms 延迟的网络损伤条件下，H1 应用层数据丢包率建议不高于 0.5%。
3. 在链路背景流量超过 80% 的情况下，雷达数据流应保持可用，允许性能下降，但不应出现长时间中断或不可恢复异常。
4. 在广播/组播压力场景下，雷达主业务通信不应被广播风暴拖垮。

## 5. 测试环境

### 5.1 硬件

1. 被测雷达：C2/H1，建议至少准备 2 台同型号设备用于对照，另准备多台用于组网测试。
2. 工业交换机：全千兆管理型交换机，建议支持端口镜像、VLAN、端口流量统计。
3. 网络损伤模拟设备：支持延迟、抖动、丢包、乱序、重复包注入。
4. 工控机或测试电脑：建议一台 Windows 10/11，一台 Ubuntu 20.04/22.04；Ubuntu 机器用于 ROS 和自动化测试。
5. 可编程直流电源：满足 C2/H1 输入电压范围和纹波注入需求。
6. 移动小车或转台：用于动态数据连续性和帧率波动测试。
7. 示波器：用于电源纹波、瞬态电压和网口相关辅助分析。
8. 网线：CAT5e/CAT6，至少包含 1 m、5 m、10 m、15 m 多种长度。

### 5.2 软件

1. Python 自动化环境：使用项目 `.venv` 虚拟环境，优先通过 `uv` 管理依赖与运行脚本。
2. Pytest：用于组织自动化用例。
3. Wireshark：用于抓包、重传、丢包、协议字段分析。
4. iperf3、Scapy、Ostinato：用于背景流量、异常流量、压力流量制造。
5. Nmap：用于开放端口扫描和基础安全检查。
6. ROS2：用于 ROS 驱动接入、点云话题验证和长稳测试。

## 6. 测试用例整理

### 6.1 HW：硬件与连接测试

HW-01 电源适应性：

1. 在规格范围内调节电压，C2 参考 9-26 V，H1 参考 12-26 V。
2. 施加 ±5% 纹波噪声。
3. 模拟瞬态电压变化。
4. 记录雷达是否重启、掉线、数据异常。

通过条件：雷达持续工作，通信连接稳定，数据无明显异常。

HW-02 接口与线缆：

1. 对网口进行 100 次插拔循环。
2. 使用 1 m、5 m、10 m、15 m 线缆分别测试。
3. 做线缆弯折、拉伸后的通信验证。

通过条件：接口无物理损伤，各线缆长度下通信稳定，无速率明显下降。

HW-03 网络基础连通性：

1. 配置雷达与测试机在同一子网。
2. 持续 ping 7 天或按项目阶段缩短为 8 小时、24 小时、72 小时分级执行。
3. 记录成功率、平均延迟、最大延迟。

通过条件：ping 成功率不低于 99.9%，平均延迟满足基础标准。

### 6.2 PROT：通信协议与配置测试

PROT-01 IP 配置与发现：

1. 验证静态 IP 配置、保存、重启后保持。
2. 若固件支持 DHCP，则验证自动获取 IP。
3. 验证上位机或协议广播发现能力。
4. 记录不同交换机环境下的发现成功率。

通过条件：IP 配置正确、可持久化，设备可被稳定发现。

PROT-02 应用层协议：

1. 发送启动数据流、停止数据流、状态查询、参数读取、参数设置等标准指令。
2. 验证响应时间、响应格式、错误码。
3. 验证点云或测距数据格式、频率、字段范围。

通过条件：协议响应正确率 100%，数据流连续，格式符合对应型号协议定义。

注意：H1/H2 系列以太网通信需优先对齐 H1E0-02A 产品说明书第 4 章和表 4-2 的 TCP 文本帧规则；若 C2/H1 实际固件使用不同端口或私有协议，需要以项目内最新协议文档和抓包结果为准。

PROT-03 协议健壮性：

1. 发送畸形、超长、截断、重复、乱序指令。
2. 模拟未登录先发业务数据、未启动先停止、重复启停等状态机异常。
3. 高频建立和断开连接。

通过条件：设备丢弃非法数据，不崩溃、不死机，正常会话能快速恢复。

PROT-04 多客户端连接：

1. 多台上位机同时连接雷达配置端口。
2. 多台上位机同时订阅数据端口。
3. 观察连接抢占、拒绝、并发输出策略。

通过条件：行为与设计一致，并形成清晰说明，例如“配置连接只允许一个，数据连接允许多个”。

### 6.3 NET：网络健壮性测试

NET-01 网络损伤：

1. 注入固定延迟：10 ms、50 ms、100 ms、200 ms。
2. 注入随机抖动：±10 ms、±30 ms。
3. 注入丢包：0.1%、0.5%、1%、3%。
4. 注入乱序包、重复包。
5. 统计应用层数据连续性、延迟、丢帧率、重连次数。

通过条件：满足压力准入标准，不因单包丢失导致整条会话不可恢复。

NET-02 异常与恢复：

1. 运行中拔插网线。
2. 制造 IP 地址冲突。
3. 重启交换机。
4. 模拟 ARP 欺骗或泛洪攻击。
5. 记录故障发生、链路恢复、应用层数据恢复的时间点。

通过条件：恢复时间满足标准，恢复后数据顺序和状态正确。

NET-03 带宽压力与背景流量：

1. 使用 iperf3 制造 80% 以上链路背景流量。
2. 同时运行雷达数据流。
3. 记录吞吐、丢包、TCP 重传、CPU/内存占用。

通过条件：主数据流保持可用，系统资源无异常飙升。

NET-04 组播和广播影响：

1. 在同一局域网持续发送高频广播包。
2. 持续发送组播流量。
3. 观察雷达通信质量和交换机端口统计。

通过条件：雷达不被广播风暴拖垮，业务数据可持续接收。

### 6.4 SYS：整机集成与动态测试

SYS-01 车载环境适应性：

1. 将雷达安装到移动小车或转台。
2. 覆盖静止、低速、中速、高速、加速、刹车、转弯、颠簸路面。
3. 记录连接状态、帧率、丢帧率、TCP 重传率。

通过条件：网络连接不中断，数据流连续，帧率波动满足 ±10%。

SYS-02 多雷达组网：

1. 多台 C2/H1 接入同一交换机。
2. 所有雷达同时输出数据。
3. 记录端口带宽、数据流标识、时间戳、丢帧率。

通过条件：各雷达数据独立，不串流，不互相明显影响。

SYS-03 多传感器共存：

1. 雷达与摄像头、毫米波雷达、GPS/IMU 等接入同一网络。
2. 同时运行所有传感器。
3. 观察带宽占用和数据稳定性。

通过条件：雷达通信正常，网络带宽分配合理，无明显性能劣化。

### 6.5 REL/SEC：可靠性与安全测试

REL-01 长期稳定性：

1. 典型负载连续运行 72 小时。
2. 满负载连续运行 72 小时。
3. 项目后期可扩展到 168 小时准入测试。
4. 记录连接中断、重连、丢帧、内存增长、CPU 占用。

通过条件：无崩溃、无死机、无明显资源泄漏，数据累计丢帧率接近 0。

REL-02 固件升级可靠性：

1. 正常升级验证。
2. 升级过程中断电后重新上电。
3. 传输损坏固件包。
4. H1 如需执行 2000 次升级压力循环，应单独准备自动化脚本和设备保护策略。

通过条件：正常升级成功，异常升级不变砖，能进入安全模式或恢复出厂固件。

SEC-01 网络安全扫描：

1. 使用 Nmap 扫描 TCP/UDP 开放端口。
2. 检查是否存在非业务端口。
3. 尝试从非授权 IP 访问服务。

通过条件：仅开放必要端口，无明显高危漏洞。

SEC-02 协议模糊测试：

1. 对开放端口发送随机、畸形、超长数据。
2. 观察设备是否崩溃、重启、泄露异常信息。
3. 记录异常包样本和设备响应。

通过条件：设备可拒绝异常输入，不影响正常业务恢复。

### 6.6 ROS 驱动测试补全

ROS-01 点云/测距话题测试：

1. 启动 ROS 驱动。
2. 检查话题是否正常发布。
3. 统计频率、消息大小、时间戳连续性。

通过条件：话题频率稳定，消息字段完整，时间戳连续。

ROS-02 取数健壮性：

1. 持续订阅数据 8 小时、24 小时、72 小时。
2. 记录丢帧、异常日志、驱动 CPU/内存占用。

通过条件：驱动不崩溃，资源占用稳定。

ROS-03 异常输入与参数测试：

1. 使用错误 IP、错误端口、错误参数启动驱动。
2. 修改帧率、坐标系、话题名等参数。

通过条件：错误提示清晰，不出现无响应或崩溃。

ROS-04 网络中断恢复：

1. 驱动运行中拔网线。
2. 重启雷达。
3. 重启交换机。

通过条件：驱动可自动重连或明确提示用户重新启动，恢复后数据正常。
